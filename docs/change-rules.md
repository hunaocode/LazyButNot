# 变更约束

## 1. 目的

本文件用于约束 AI coding 或多人协作时对关键模块的修改方式，避免“修一个 bug，带出一串隐性回归”。

适用范围：
- 提醒系统
- AlarmKit
- 倒计时生命周期
- 目标打卡后的状态流转
- 首页目标卡片交互
- 主题与页面视觉一致性

## 2. 改动前必须先回答的问题

任何涉及提醒逻辑的改动，在动代码前必须先明确：
- 当前 bug 的根因是什么
- 这次修改只想修哪个行为
- 哪些既有规则必须保持不变
- 这次修改会影响哪些模块
- 改完后需要验证哪些回归场景

如果以上 5 点说不清，不应直接修改代码。

## 3. 提醒模块的硬约束

修改 `NotificationManager` 时，默认必须保持以下规则不变，除非任务明确要求修改它们：
- 当前提醒模型是 fixed-date window
- 当前 `rollingScheduleHorizonDays = 7`
- `weeklyFixedDays` 搜索窗口是 `7 * 3`
- 当天已完成时，重建窗口默认不包含今天
- 普通前后台切换不自动触发 `scheduleAll`
- AlarmKit 仅用于监督提醒

若确实要修改以上任一条，必须同时：
- 更新 `PRODUCT_REQUIREMENTS.md`
- 更新 `reminder-engine.md`
- 重新过 `regression-checklist.md`

## 4. 不允许的改法

以下做法默认禁止：
- 未经确认就把 fixed-date 改回 repeating
- 未经确认扩大提醒窗口天数
- 顺手改前后台重排策略
- 只修 `daily`，不检查 `weeklyFixedDays` 和 `weeklyCount`
- 只改 schedule，不检查 remove/cancel 路径
- 只看编译通过，不做回归检查

## 5. 推荐任务拆分方式

涉及关键逻辑时，推荐按以下顺序执行：

1. 先分析
- 根因
- 影响面
- 不能破坏的规则

2. 再修改
- 只动必要文件
- 不做顺手重构

3. 再验证
- 编译
- 日志
- 回归清单

4. 最后更新文档
- PRD
- 模块规则文档

## 6. 对 AI 的推荐指令模板

建议在每次复杂修改前，先给模型这样的指令：

```text
先不要改代码。
先给我：
1. 根因
2. 影响范围
3. 必须保持不变的现有规则
4. 回归验证清单
确认后再修改。
```

如果是做 code review，建议使用：

```text
请按 code review 方式检查这次未暂存修改：
- 先列 findings，按严重度排序
- 只讲 bug / 回归 / 逻辑不一致
- 给出文件和行号
- 最后再讲开放问题和测试缺口
```

## 7. 哪些内容必须沉淀进仓库

以下内容不能只存在聊天记录里：
- 当前提醒模型
- 当前窗口大小
- 每种周期的调度规则
- 打卡完成后的取消规则
- 前后台触发策略
- 回归清单
- 首页卡片哪些区域可进入详情、哪些区域只执行原地操作
- 已完成与未完成卡片在视觉和交互上的差异
- 主题当前覆盖到哪些页面、哪些控件会跟随主题变化

只要某条规则被重复强调两次以上，就应该沉淀进仓库文档。

## 8. UI 改动的额外约束

涉及首页卡片、目标列表、统计页、设置页或主题系统时，默认还要保持以下规则不变，除非需求明确要求修改：
- 首页卡片除明确按钮区外，其余内容区应可进入目标详情
- 未完成卡片与已完成卡片必须有明显的状态区分，不能只靠文案颜色微调
- 已完成卡片不应继续保留会误导用户的重复操作按钮
- 主题切换后，不应只改首页和目标页，统计页与设置页也要检查是否仍保持视觉一致
- 若需求只说“改配色 / 做主题适配”，默认不允许顺手改 `padding`、`spacing`、`listRowInsets`、`listStyle`、`frame`、卡片分组数量或层级
- 页面卡片布局必须拆成 3 个独立层次来控制：
- 卡片内容内边距
- 卡片到页面两侧的外边距
- 模块与模块、分组与分组之间的间距
- 修改其中任意一层时，不得隐式改变另外两层
- 对 SwiftUI `List` 页，必须先明确“卡片视觉”归属在哪一层：
- 若卡片画在内容自身的 `.background(...)` 上，则 row 背景必须保持 `Color.clear`
- 若卡片画在 `listRowBackground(...)` 上，则要明确这会直接影响 row inset、模块间距和分组留白
- 不允许混用两种卡片承载方式后再靠试错修间距
- 设置页、统计页这类分模块页面，默认规则是“一个业务模块 = 一张卡”；除非需求明确要求拆开，否则不能把一个模块拆成多张小卡
- 外观主题这类“分组中的分组”，分组卡片之间的留白应单独控制，不能通过改整个页面的默认 list 样式去碰运气
- `List` 从 `.insetGrouped` 改成 `.plain`、或反过来，属于布局层级变更，不属于纯配色修改；若需求只要求主题适配，默认不改

### 8.1 UI 修改前的基线记录

涉及卡片页或主题页时，修改前至少先记录以下 4 项当前基线：
- 模块卡片到页面左右的距离是多少
- 卡片内部内容的内边距是多少
- 模块与模块之间的垂直间距是多少
- 分组标题与卡片之间的距离是多少

若这 4 项没有先说清，后续很容易出现“颜色改好了，但布局节奏被顺手带坏”的问题。

### 8.2 纯主题适配的推荐改法

如果需求本质上是“只改颜色，不改排版”，默认应按以下顺序执行：

1. 先锁定不允许动的布局属性
- `padding`
- `spacing`
- `listRowInsets`
- `listStyle`
- 卡片数量和分组层级

2. 再替换颜色来源
- `foregroundStyle`
- `fill`
- `stroke`
- `shadow`
- 主题标签和强调色

3. 最后只做回归验证
- 颜色是否跟随主题
- 布局节奏是否完全保持原状

若以上任一条被调整，必须同步更新：
- `docs/PRODUCT_REQUIREMENTS.md`
- `docs/regression-checklist.md`
